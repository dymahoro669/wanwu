# 万物 App 项目经验记录

## 反思：为何未能提前发现这些明显问题

### 问题回顾
用户测试发现 9 个前端问题，均为实现与原型不一致：
1. 启动页视频应一开始就在背景播放，而非等2秒后才单独展示
2. 启动后应进入登录页，而非直接进入首页
3. 首页字体与原型不一致
4. 首页布局颜色与原型不一致
5. 导航栏选中状态应为圆角（原型使用 `rounded-full`），而非直角
6. 数据TAB背景是动画而非原型中的静态效果
7. TAB页链接错误，"我的"页面应指向个人中心
8. 资产卡片无详情页链接
9. 页面宽度应限制为 448px（原型使用 `max-w-screen-md` ≈ 448px）

### 问题根源分析

**设计阶段缺失**
- 没有在实现前与用户确认"启动→登录→首页"的完整流程
- 原型页面之间的跳转关系未明确

**开发阶段问题**
- 页面宽度约束：`max-w-screen-md` 被忽略，导致占满全屏
- 字体：原型使用 `Noto Serif SC` (font-headline-md)，代码未统一应用
- 颜色变量：原型定义完整颜色系统（surface-container, primary-container 等），实现时未完全采用
- 圆角：导航栏激活状态原型用 `rounded-full py-1.5 px-8`，实现时缺少

**验证阶段问题**
- 仅依赖代码生成，未进行实际设备/模拟器测试
- Playwright 测试未覆盖导航流程、字体渲染、颜色值等细节

### 经验教训

1. **明确页面跳转流程**：在开始实现前，与用户确认完整的用户流程图
2. **像素级还原**：检查每一个 CSS 属性，包括圆角、内边距、最大宽度
3. **使用实际设备验证**：Web 预览与移动端实际表现可能有差异
4. **检查所有页面链接**：确保页面映射正确（profile → _2/个人中心）
5. **组件样式细节**：选中状态、hover 效果等都要与原型一致

### 后续行动
- 每次实现新页面时，对照原型检查所有视觉属性
- 完成后在实际设备或模拟器上验证

## 2026/04/22 经验教训

### 问题
修改登录页修改了5次，每次只修一个问题（字号、字体、颜色、图标等），导致反复返工。

### 根因
没有先完整读取原型文件并记录所有关键样式，而是边改边看、边看边改。

### 行动
**修改任何页面之前，必须先完整读取原型 HTML，将所有关键样式（字号、字体、颜色、间距、图标规格）列成清单，一次性修改到位。**
